02 / 07
the approach

Spec before code

Before writing a single component, I used Claude Code to build the constraints of the project first. Not just a product spec, but the roles that would normally exist around a product like this.

Senior software engineer
complexity must earn its place
Senior prompt / AI engineer
output must be consistent and parseable
Employer branding specialist
never sound like a robot
Performance marketer
full value first, no friction gates
Product designer
mobile viewport, every state covered
Recruitment copywriter
UI copy should sound human
03 / 07
prompt 1

Defining the product first

actual prompt
"I'm building the Job Ad Doctor from the Teamtailor brief. Before we write any code, I want a single product definition document that covers: who the user types are, the root cause of bad job ads, what good looks like, the weakness categories, the user journey, error states, and success metrics. This becomes the source of truth."

output: docs/project-owner-expectations.md

Nothing was built until this existed. Every role and technical decision was held against it.

04 / 07
prompt 2

Generating the specialist roles

actual prompt
"Now create the specialist roles around this product. For each role, define: the lens they use to evaluate decisions, their quality bar, and the decisions they own. These will be used as constraints for implementation."

This gave me six different lenses before a line of code: engineering simplicity, prompt reliability, UX clarity, employer-brand authenticity, conversion honesty, UI language quality. Later build decisions could be checked against all of them.

05 / 07
prompt 3

Architectural planning against the spec

actual prompt
"Using the product owner doc and all specialist role docs as constraints, create the build plan. Every architectural decision must be defensible on trust, scalability, latency, cost, and user experience."
  • Two LLM calls: better latency, clearer user feedback loop
  • Strict JSON output so the UI can trust and validate it
  • Server-side API routes: keys never exposed client-side
  • Rate limiting specced from day one, stubbed intentionally
  • Prefetch rewrite in background while user reads diagnosis
06 / 07
reflection

Where it helped vs. got in the way

helped

Claude Code never drifted because it had the product definition and role quality bars as context. It did not suggest severity scores (spec said present/absent only). It did not suggest auth (spec said zero friction). Constraint docs did the guardrail work.

got in the way

When given too much freedom it reached for abstraction. It proposed a state machine for rewrite prefetching. I simplified to two refs: one cache ref, one promise ref. Human judgment kept complexity proportional to the problem.

07 / 07

The takeaway

define constraints
->
pressure-test with roles
->
execute with AI
->
apply human judgment

The biggest value was not AI writing code. It was building the right constraints first, using multiple expert lenses, and then using AI to execute inside those boundaries.

AI is a fast junior who occasionally hallucinates the docs. The value is in knowing exactly what to ask for and having the specs to keep it honest.
